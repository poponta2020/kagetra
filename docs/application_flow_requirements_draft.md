# 申込フロー管理機能 要件定義（ドラフト）

> 作成: 2026-02-17
> ステータス: **ヒアリング中（未完成）**
> 次回: 各ステップの詳細設計を詰める

---

## 目的

大会申込の**締切後の事務処理フロー**を可視化・管理する機能を追加する。

現状では、大会詳細画面で参加者名と級が表示されるのみで、以下の課題がある：
- 申込書作成や主催者への送付など、事務作業の進捗が分からない
- 抽選がある大会で当選/落選の管理ができない
- 参加費の建て替え金額が記録されていない
- 管理者が「次に何をすべきか」が不明確

この機能により、**大会ごとの事務処理の進行状況を一目で把握し、次のアクションを明確にする**。

---

## スコープ

### 対象
- 締切後かつ参加者がいる大会の事務処理フロー管理
- 管理者向けの進捗管理画面
- 一般ユーザー向けの自分が関連する大会の状況確認画面

### 対象外
- 締切前の大会申込機能（既存機能で対応済み）
- 大会結果の入力・管理（既存の `result` 機能で対応済み）
- リアルタイム通知機能（既存のWeb Push/LINE通知で対応済み）

### 1. 画面構成

| 画面 | 対象ユーザー | 表示内容 |
|------|------------|---------|
| **申込フロー一覧画面** | 管理者 | 締切後かつ参加者がいる大会で、開催日前の大会一覧 |
| 同上 | 一般ユーザー | 自分が申し込んだ大会のうち、フロー管理対象の大会のみ |
| **申込フロー詳細画面** | 管理者・一般ユーザー | 大会の現在のステップ、次のタスク、参加者情報 |

**遷移元:**
- 独立した「申込管理」メニュー（トップバーに追加）

**画面デザインのイメージ:**
- 画面左端に縦向きのフロー図（丸と直線）
- 現在のステップまで色が付いている
- 現在のステップで実施すべきタスクが表示される
- タスク完了後、次のステップに進むボタン

---

### 2. 申込フローのステップ（基本）

| # | ステップ名 | 説明 | 主なタスク |
|---|-----------|------|-----------|
| 1 | 会内締切 | 景虎での申込締切を過ぎた状態 | 参加者確定 |
| 2 | 大会申込 | 申込書を作成して主催者に送付 | 申込書作成・送付 |
| 3 | 返信待ち | 主催者からの受理連絡待ち | 返信確認 |
| 4 | 抽選結果待ち | 抽選がある大会のみ | 当選/落選/キャンセル待ちの確認 |
| 5 | 参加費支払い | 会が建て替えて支払い | 支払い金額確認・振込実施 |
| 6 | 完了 | 大会当日を迎えた状態 | - |

**条件分岐:**
- 抽選なしの大会 → ステップ3,4をスキップ
- 現地払いの大会 → ステップ5をスキップ
- ステップの有無は大会ごとに設定可能か？（要確認）

### 3. 表示したい情報

#### 申込フロー一覧画面

| 項目 | 表示内容 |
|------|---------|
| 大会名 | 大会の名称 |
| 開催日 | 大会の開催日 |
| 現在のステップ | 「大会申込」「返信待ち」など |
| 参加者数 | 「12名」など |
| 次のアクション | 「申込書を作成」など |

#### 申込フロー詳細画面

| 項目 | 表示内容 |
|------|---------|
| 大会基本情報 | 大会名、開催日、締切日 |
| フロー進捗図 | 縦向きの図で現在地を表示 |
| 参加者一覧（基本） | 級別、名前（漢字） |
| 参加者詳細（申込書用） | 名前（漢字・ふりがな）、級、段位、今年度の公認大会出場回数 |
| 次のタスク | 「申込書を作成する」などのアクション |
| 期限表示 | 返信期限、支払期限など |

**参加者詳細の表示条件:**
- 「申込書作成用データ確認」ボタンを押下したときのみ表示
- 常時表示すると画面が煩雑になるため

### 4. 申込書作成機能

| 機能 | 説明 |
|------|------|
| データ表示 | 画面上に参加者情報を一覧表示 |

**技術的検討事項:**

- CSVのエンコーディング（UTF-8 BOM付き or Shift_JIS）

---

### 5. 参加費支払い管理

**基本方針:**
- 会が全員分を建て替えて主催者に振り込む
- 後日（半年に1回など）個人から集金する

**管理したい情報:**
- 個人別の参加費金額
- 合計金額
- 会が建て替えた金額の記録（後で個人から集金するため）
- 主催者への支払い状況（支払済み/未支払い）

**画面表示:**
- 「誰がいくら支払うのか」の一覧
- 合計金額
- 支払期限と振込先（大会情報に含まれる）

---

### 6. 抽選結果管理

**管理レベル:**
- 個別管理（参加者ごとに当選/落選/キャンセル待ちを設定）

**状態:**
- 当選
- 落選
- キャンセル待ち（落選後も追跡）

**キャンセル待ちの運用:**
- キャンセル待ちの人が繰り上がった場合、ステータスを「当選」に変更
- 落選確定の場合は「落選」のまま

## 技術的検討事項（後で詰める）

### データベース設計

**新規テーブル案:**

```ruby
# event_application_flows
create_table(:event_application_flows) do
  primary_key :id
  foreign_key :event_id, :events, null: false
  Integer :current_step, null: false, default: 1  # 1〜6
  TrueClass :has_lottery, default: false         # 抽選あり
  String :payment_method, default: 'advance'     # 'advance', 'onsite'
  Date :application_sent_at                      # 申込書送付日
  Date :response_deadline                        # 返信期限
  Date :payment_deadline                         # 支払期限
  Integer :total_fee                             # 合計参加費
  TrueClass :payment_completed, default: false   # 支払済み
  Text :memo                                     # メモ欄
  DateTime :created_at
  DateTime :updated_at
end

# event_user_application_statuses（参加者ごとの状態）
create_table(:event_user_application_statuses) do
  primary_key :id
  foreign_key :event_user_choice_id, :event_user_choices, null: false
  String :lottery_status  # 'pending', 'won', 'lost', 'waiting'
  Integer :fee            # 個人の参加費
  TrueClass :fee_paid, default: false  # 個人から集金済み
  DateTime :created_at
  DateTime :updated_at
end
```

### フロントエンド実装

- Backbone.js ベースで実装（既存と統一）
- 縦向きステッパーのCSS/HTML
- CSV出力機能（既存の大会結果Excel出力を参考に）

### バックエンド実装

- 新規ルート: `/routes/application_flow.rb`
- 新規モデル: `/models/application_flow.rb`
- APIエンドポイント:
  - `GET /api/application_flow/list` - フロー一覧
  - `GET /api/application_flow/detail/:event_id` - フロー詳細
  - `PUT /api/application_flow/progress/:event_id` - ステップ進行
  - `GET /api/application_flow/export_csv/:event_id` - CSV出力
